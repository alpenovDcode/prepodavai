# Tutor Exchange — Smoke checklist для беты

Ручной e2e для 3 ролей:
- **АДМИН** (`/check/prrv/admin/*`)
- **A** — репетитор-заказчик (у него ученик, он публикует заявку)
- **B** — репетитор-исполнитель (ищет заявки, откликается)

Перед началом убедиться, что в `.env` у пользователя А и Б `ADMIN_USER_IDS` не входит их id (иначе Guard бесконтрольно пропускает всё).

## 0. Предпосылки

- [ ] БД содержит миграцию `20260702120000_add_tutor_exchange` (6 таблиц: `Lead`, `LeadDialog`, `LeadMessage`, `ViolationReport`, `TutorRating`, `TutorMarketProfile`).
- [ ] Флаг `tools.tutor_exchange.enabled` записан в `SystemSetting` (или отсутствует — дефолт `false`).
- [ ] Оба тестовых аккаунта А и Б залогинены в разных браузерных сессиях.

## 1. Флаг + видимость (этап 1)

**АДМИН** → `/check/prrv/admin/tools`
- [ ] Тумблер `tutor_exchange` = OFF. Save.
- [ ] У А, Б в сайдбаре секция «Биржа» **не видна**.
- [ ] У А `/dashboard/leads` → 503 с текстом «Биржа лидов скоро откроется…».

**АДМИН** ставит тумблер ON. Save.
- [ ] У А, Б в сайдбаре появилась секция «Биржа» → «Заявки», «Диалоги» (F5).
- [ ] `/dashboard/leads` открывается.

## 2. Публикация заявки (этап 2)

**А** → `/dashboard/leads/new`
- [ ] Шаг 1: выбрать тип `COMMISSION` (комиссия ≥100 ₽).
- [ ] Шаг 2: subject «Математика», grade «10 класс», format `OFFLINE`, city «Москва», description ≥30 символов, studentContact «+7 999 111-22-33», price 1500.
- [ ] Шаг 3: превью → создать.
- [ ] Редирект в `/dashboard/leads/:id` → карточка с деталями, контакт видим (creator).

**Б** → `/dashboard/leads`
- [ ] Заявку А видит в ленте.
- [ ] В фильтре по subject «математика» (нижний регистр) — заявка найдена.
- [ ] Открывает `/dashboard/leads/:id` — карточка есть, контакт скрыт (`Скрыт — станет виден после закрытия сделки`).

## 3. Отклик + чат + модерация (этап 3)

**Б** нажимает «Откликнуться».
- [ ] Создаётся диалог, редирект в `/dashboard/dialogs/:id`, статус `Открыт`.

**А** заходит в `/dashboard/dialogs` — новый диалог виден в списке.

**Б** пишет «Привет, готов взяться»
- [ ] Сообщение появилось у обоих (polling 3с).

**Б** пишет «Позвоните +7 999 111-22-33»
- [ ] Сообщение помечено `помечено модерацией`, следом системное предупреждение с эмоджи ⚠️.

**А** нажимает **«Сообщить о нарушении»** → описывает → отправить.
- [ ] Появляется alert «Жалоба отправлена…».

**АДМИН** → `/check/prrv/admin/violations`
- [ ] Видит жалобу А. Кнопки «Отклонить» / «Удовлетворить» работают.

## 4. State-machine (этап 4)

**А** ставит `«Назначить пробный урок»` → указывает ссылку `https://zoom.us/j/123` → назначить.
- [ ] Статус → `Пробный урок`. У Б виден trial-link.

**Б** «Пробный прошёл успешно».
- [ ] Статус → `Оплата комиссии`. Появился countdown до `paymentDeadline` (3 дня).

**Б** «Я отправил оплату».
- [ ] Кнопка disabled → «Оплата отмечена». У А появилась кнопка `«Подтвердить получение оплаты»`.

**А** «Подтвердить получение оплаты».
- [ ] Статус → `Сделка закрыта`. Контакт ученика открыт у обоих.

## 5. Рейтинги (этап 5)

**А** нажимает «Оценить сделку» → 5 звёзд + комментарий → отправить.
- [ ] Кнопка → «Оценка отправлена».

**Б** тоже оценивает.
- [ ] Оба в `/dashboard/tutor/<id-другого>` видят: рейтинг ⭐, счётчик сделок = 1, отзыв.

## 6. Уведомления

**А** и **Б** проверяют bell / `/api/notifications/*`.
- [ ] Есть `tutor_exchange.*` события за всю цепочку (dialog_created, trial_scheduled, trial_result, payment_reported, payment_confirmed, rating_received).

Если у А/Б привязан BotUser с `telegramId`:
- [ ] Приходят короткие сообщения с deep-link `/dashboard/dialogs/:id` для событий дальнего сектора (dialog_created, trial_result, payment_reported/confirmed).

## 7. Негативные сценарии

- [ ] У Б есть просроченный `PAYMENT_PENDING` — при повторном отклике на новую заявку получает 400 `OverduePayment`.
- [ ] Б создаёт 5 активных диалогов, на 6-й отклик — 400 `LimitReached`.
- [ ] `А` пытается открыть чужой диалог (не участник) — 403.
- [ ] `Б` пытается назначить пробный урок — 403 (только creator).
- [ ] `Cancel` из `OPEN` возвращает заявку в ленту с `ACTIVE`.
- [ ] `TRIAL_FAIL` из `TRIAL_PENDING` возвращает заявку в ленту с `ACTIVE`.

## 8. Cron просрочки

- [ ] Из `psql`: `UPDATE lead_dialogs SET payment_deadline = NOW() - INTERVAL '1 hour', payment_overdue_notified_at = NULL WHERE id = '<dialogId>';`
- [ ] Дождаться выстрела `PaymentOverdueJob` (раз в час) **или** вызвать вручную из ноды.
- [ ] В логах: `Payment overdue marked: 1 dialog(s)`. У А — уведомление `payment_overdue`.

## Rollout

1. Включить флаг из `/check/prrv/admin/tools`. Убедиться, что `message` пустое (или дефолтное) — оно скрывается на UI при `enabled=true`.
2. Постучаться в мессенджер бета-репетиторам (3–5 человек) — «биржа открыта, попробуйте».
3. Наблюдать неделю: логи `DialogActionsService.transition ...`, `/admin/tutor-exchange/violations`, репорты в поддержку.
4. По итогам — либо расширяем на всех, либо возвращаем флаг в OFF, диагностируем.
